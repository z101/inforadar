# Архитектура приложения "Информационный радар"

## Обзор

Приложение построено по модульному принципу с четким разделением ответственности между компонентами. Это упрощает добавление новых источников данных и изменение логики работы отдельных частей системы.

Основной поток данных при сборе выглядит следующим образом:

`CLI -> Core Engine -> Provider -> Storage`

## Ключевые компоненты

1.  **CLI (`main.py`)**
    *   **Технологии:** `Typer`, `Rich`.
    *   **Ответственность:** Пользовательский интерфейс. Принимает команды (`fetch`, `list`), инициализирует и вызывает `Core Engine`.

2.  **Core Engine (`core.py`)**
    *   **Ответственность:** Оркестрация. Является связующим звеном между CLI, провайдерами и хранилищем. Запускает процесс сбора данных по всем настроенным провайдерам и передает полученные статьи в хранилище.

3.  **Providers (`providers/`)**
    *   **Ответственность:** Сбор и обогащение данных из конкретного источника.
    *   Каждый провайдер (например, `HabrProvider`) реализует логику получения статей (RSS, скрапинг) для своего источника.

4.  **Storage (`storage.py`)**
    *   **Технологии:** `SQLAlchemy`.
    *   **Ответственность:** Взаимодействие с базой данных (`SQLite`). Предоставляет API для сохранения, получения и обновления статей, скрывая детали реализации БД от остального приложения.

5.  **Models (`models.py`)**
    *   **Ответственность:** Определение структур данных. `Article` - основная модель данных, описанная с помощью ORM `SQLAlchemy`.

6.  **Configuration (`config.yml`)**
    *   **ОтветственGность:** Конфигурация приложения. Позволяет пользователю настраивать источники данных (например, хабы Habr) без изменения кода.
