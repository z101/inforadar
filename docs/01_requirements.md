# Требования к MVP "Информационный радар"

## Цель MVP

Продемонстрировать ключевую ценность "Информационного радара": **быстрый и удобный разбор большого потока статей из кастомных источников (Habr) для экономии времени пользователя.** MVP будет представлять собой полнофункциональное консольное приложение, реализующее цикл "Сбор -> Обогащение -> Фильтрация -> Интерактивный разбор".

---

## Функциональные требования (ФТ):

- **ФТ1: Конфигурация через файл:** Приложение инициализируется из файла `config.yml`, где жестко задан список интересующих хабов Habr и правила автоматической фильтрации (минимальный рейтинг, ключевые слова для исключения).
- **ФТ2: Двухэтапный сбор данных:** Будет реализован кастомный `HabrProvider`, который:
    - а) Обнаруживает новые статьи через RSS-фиды указанных хабов.
    - б) Обогащает каждую новую статью, скрапя ее HTML-страницу для получения рейтинга, просмотров, времени чтения и т.д.
- **ФТ3: Хранение данных:** Данные сохраняются в локальной базе данных SQLite (`inforadar.db`). Дубликаты предотвращаются проверкой `guid` статьи. Вся дополнительная метаинформация (рейтинг, просмотры) хранится в JSON-колонке.
- **ФТ4: Автоматическая фильтрация:** В процессе сбора данных статьи, не соответствующие правилам в `config.yml`, автоматически отбрасываются и не попадают в список для разбора.
- **ФТ5: Интерактивный разбор:** Основной режим `inforadar list --interactive`, который позволяет в формате "одна статья - один экран" быстро принимать решения с помощью одноклавишных команд (`open`, `skip`, `interesting`, `discard`).
- **ФT6: Сохранение состояния:** Приложение должно помнить, какие статьи новые, какие прочитаны, а какие отложены как "интересные", между запусками.

## Нефункциональные требования (НФТ):

- **НФТ1: Этичный скрапинг:** Сборщик должен соблюдать приличия: устанавливать `User-Agent`, делать паузы между запросами к HTML-страницам (rate limiting), чтобы избежать бана.
- **НФТ2: Комфортный CLI:** Интерфейс должен быть отзывчивым. Длительные операции (сбор данных) должны сопровождаться наглядным прогресс-баром. Вывод информации должен быть структурирован и приятен глазу.
- **НФТ3: "One-shot" операции:** Все команды должны быть самодостаточными и не требовать от пользователя дополнительных уточнений в диалоговом режиме (кроме интерактивного разбора).
